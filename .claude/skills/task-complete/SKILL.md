# 任务完成

标记子任务为已完成，按 audit-scope 执行审计，自动修复问题并循环重审，直到全部通过后联动更新父任务，等用户确认后提交。

## 参数

- `子任务路径`（可选）：子任务目录路径，默认根据当前上下文确定

## 用法示例

```
/task-complete
/task-complete .process/task/task-phase4/subtask_1
```

## 执行步骤

### 1. 确定任务路径

- 如果提供了参数，使用指定路径
- 如果没有参数，根据当前上下文确定正在执行的子任务路径

### 2. 读取任务信息

读取子任务的 `task.md`：
- 任务目标和验收标准
- **audit-scope**：关联的 audit agent 列表
- **输出类型**：代码 / 文档 / 配置

读取子任务的 `progress.md`：
- 当前状态和进度

### 3. 审计循环（最多 5 轮）

#### 3.1 暂存变更

```
git add .
```

#### 3.2 按 audit-scope 执行审计

读取 `task.md` 中定义的 `audit-scope`，调用对应的 audit agent：

- **串行**逐一运行每个 audit agent（禁止并行），等上一个完成后再启动下一个
- 收集所有审计结果，分类为：
  - **❌ 冲突**（必须修复）
  - **⚠️ 警告**（应该修复）
  - **💡 建议**（推荐修复）

#### 3.3 安全检查（所有提交）

对所有提交执行：
```
gitleaks detect -v
```

#### 3.4 代码输出额外检查

如果输出类型为"代码"，额外执行：
```
go fmt ./...
go vet ./...
golangci-lint run ./...
go test ./...
```

#### 3.5 自动修复所有问题

**目标：修复审计中发现的所有冲突、警告和建议项，不留遗留。不能以"可接受"、"低风险"、"等价"为由跳过。**

按优先级处理：

1. **冲突项（❌）**：立即修复，这些是明确违反规则的问题
2. **警告项（⚠️）**：必须修复，不能跳过
3. **建议项（💡）**：尽量修复。只有以下情况可跳过：
   - 修复方案涉及后续子任务的设计决策（记录原因，后续处理）
   - 修复需要用户确认需求范围（停下来问用户）

**遇到不确定的情况时，停下来向用户说明问题并提问**。以下情况需要停下：
- 修复方案涉及设计决策（需要用户确认设计意图）
- 修复方案涉及需求变更（需要用户确认需求范围）
- 不确定如何修复（多种方案、无法判断优劣）
- 修复可能引入新风险（大范围重构、API 变更等）

**不需要停下的情况（直接修复）**：
- 代码风格、命名、注释等机械性修复
- 补充缺失的测试用例
- 明确的 bug 或安全问题
- 代码对称性、一致性改进
- 性能优化（如果能证明有实际影响）
- 文档补充
- 消除冗余代码
- `*new(T)` → `var zero T` 等意图更明确的写法

**禁止的行为**：
- 以"防御性编码"为由保留冗余代码
- 以"可接受"为由跳过审计建议
- 以"等价编译"为由拒绝改进写法
- 以"正确性无问题"为由忽略性能建议

#### 3.6 循环判定

- **所有 agent 全部 PASS，无冲突/警告/建议** → 进入步骤 4
- **仍有问题但在本轮做了修复** → 回到 3.1 重新审计
- **已重试 5 轮仍未通过** → 停止，输出所有遗留问题，等待用户指令

### 4. 更新子任务状态

在子任务 `progress.md` 中追加。**审计结果必须逐条列出每个 audit agent 的结论，不得合并为顶层汇总**：

```markdown
## [YYYY-MM-DD HH:MM] 任务完成

**状态**: 已完成

**审计结果**:
- ✅ audit-agent-1: PASS
- ✅ audit-agent-2: PASS
- ✅ gitleaks: no leaks
- ✅ go test ./...: all pass
- ✅ golangci-lint: 0 issues
- ...

**总结**: <简要总结完成内容>
```

### 5. 联动更新父任务

找到父任务目录（上级目录中的 `task.md`），在父任务 `progress.md` 中追加：

```markdown
### [YYYY-MM-DD HH:MM] 子任务更新

**子任务**: <子任务名>
**状态**: 已完成
**摘要**: <子任务完成摘要>
```

### 6. 更新 CLAUDE.md 项目结构段（如有变化）

### 7. 输出审计摘要，等待用户确认

输出：
- 子任务路径
- 审计结果摘要（含修复轮次）
- 父任务更新状态

**停止，等待用户确认**。用户确认后执行步骤 8。

### 8. Git 提交（用户确认后，最多 5 次）

**禁止在用户明确确认审计结果前执行任何 git 操作（add/commit/push）。**

1. `git add .`
2. 使用 `printf 'subject\n\n- detail\n' | git commit -F -` 提交（**禁止写入临时文件**）
   - **commit message 必须使用英文**
3. 通过 pre-commit hooks，失败则修复后重试
4. 已重试 5 次仍失败 → 停止，等待用户确认
5. `git push`

## 注意事项

- 所有更新使用追加模式，不覆盖已有内容
- 一个子任务一次提交，不跨任务合并
- 审计必须基于 `task.md` 中的 audit-scope，不要自行猜测需要哪些审计
- 如果 task.md 中没有定义 audit-scope，则使用 audit-router agent 根据变更内容自动路由
- 审计循环中，不确定的修复必须停下来问用户，宁可多问一次也不要做错误假设
